# 수정지시서 001: import 경로 통일 + react-query 의존성 + 함수명 케이싱 + 출력 폴더 구조 개선

## 배경

fullend orchestrator가 STML 생성 TSX + 자체 glue-gen (api.ts, App.tsx, package.json 등)을 조합하여 프론트엔드를 구성한다.
현재 STML 생성 코드와 fullend glue-gen 코드 사이에 규약 불일치가 있다.

---

## 1. API import 경로 불일치

### 문제

STML generator가 생성하는 TSX 파일:
```typescript
import { api } from '@/lib/api'
```

fullend glue-gen이 생성하는 api.ts 위치:
```
frontend/src/api.ts
```

`@/lib/api` 경로는 Next.js의 path alias를 전제하지만, fullend는 Vite + React 구조를 사용한다.
Vite에서 `@/` alias를 설정하더라도 파일이 `src/api.ts`에 있으므로 `@/api`가 되어야 한다.

### 수정 제안

import 경로를 설정 가능하게 만들거나 상대 경로로 변경한다:

**방법 1: 설정 가능한 import 경로 (권장)**

```go
type GenerateOptions struct {
    APIImportPath string // 기본값: "@/lib/api"
}
```

fullend가 호출 시:
```go
stmlgenerator.Generate(pages, specsDir, outDir, stmlgenerator.GenerateOptions{
    APIImportPath: "../api",  // 또는 "@/api"
})
```

**방법 2: 상대 경로**

TSX 파일이 `frontend/` 직하에, api.ts가 `frontend/src/`에 있으므로:
```typescript
import { api } from '../src/api'  // 또는 동일 폴더면 './api'
```

### 구현 위치

- `generator/imports.go:130` — `'@/lib/api'` 하드코딩 부분

## 2. @tanstack/react-query 의존성 누락

### 문제

STML이 생성하는 TSX는 `@tanstack/react-query`의 `useQuery`, `useMutation`, `useQueryClient`를 사용한다.
그러나 STML은 package.json을 생성하지 않으므로, fullend glue-gen이 package.json을 만들 때 이 의존성을 알아야 한다.

fullend가 현재 생성하는 package.json:
```json
{
  "dependencies": {
    "react": "^18",
    "react-dom": "^18",
    "react-router-dom": "^6"
  }
}
```

누락된 의존성:
- `@tanstack/react-query` — useQuery, useMutation
- `react-hook-form` — useForm

### 수정 제안

**방법 1: STML generator가 필요 의존성 목록을 반환 (권장)**

```go
type GenerateResult struct {
    Pages        int
    Dependencies map[string]string // 패키지명 → 버전 범위
}

func Generate(pages []parser.PageSpec, specsDir, outDir string) (*GenerateResult, error) {
    result := &GenerateResult{
        Pages: len(pages),
        Dependencies: map[string]string{
            "@tanstack/react-query": "^5",
            "react-hook-form":       "^7",
        },
    }
    // ... 기존 생성 로직 ...
    return result, nil
}
```

fullend가 이 목록을 package.json에 병합한다.

**방법 2: fullend가 하드코딩**

fullend glue-gen이 STML이 있으면 무조건 `@tanstack/react-query`, `react-hook-form`을 추가한다.
단순하지만 STML 의존성이 바뀌면 fullend도 수정해야 한다.

### 구현 위치 (방법 1)

- `generator/generate.go` — 반환 타입 변경, 의존성 수집
- `generator/imports.go` — 사용된 import 기반으로 의존성 결정

## 3. API 함수명 케이싱 불일치

### 문제

STML이 생성하는 API 호출:
```typescript
api.ListCourses({ page, limit, sortBy, sortDir })
api.CreateReservation(data)
```

fullend glue-gen이 생성하는 api.ts:
```typescript
export async function listCourses(params?: Record<string, string>) { ... }
export async function createReservation(body?: Record<string, unknown>) { ... }
```

**불일치:** STML은 `PascalCase` (`api.ListCourses`), fullend는 `camelCase` (`listCourses`).

### 수정 제안

**방법 1: STML이 camelCase로 변경**

```typescript
// 변경 전
api.ListCourses(...)

// 변경 후
api.listCourses(...)
```

**방법 2: fullend가 PascalCase로 변경**

fullend glue-gen의 api.ts 함수명을 operationId 그대로 PascalCase로 생성.

**방법 3: api 객체로 감싸기 (권장)**

STML이 `api.ListCourses()`로 호출하므로, fullend가 api.ts를 namespace 객체로 export:

```typescript
// fullend가 생성하는 api.ts
async function listCourses(...) { ... }
async function createCourse(...) { ... }

export const api = {
    ListCourses: listCourses,
    CreateCourse: createCourse,
    // ...
}
```

이렇게 하면 STML 코드 변경 없이 fullend만 수정하면 된다.

### 권장

**방법 3** — fullend 측에서 해결. STML은 변경 불필요.
단, STML이 `import { api } from '...'` 형태의 named import를 사용하므로, fullend가 api 객체를 named export하면 호환된다.

## 4. 출력 폴더 구조 개선

### 현재 구조

```
outDir/                          # frontendDir (= artifactsDir/frontend)
├── login-page.tsx               # STML이 직접 생성
├── course-list-page.tsx
├── my-enrollments-page.tsx
├── src/                         # fullend glue-gen이 생성
│   ├── App.tsx
│   ├── api.ts
│   └── main.tsx
├── package.json
├── vite.config.ts
└── index.html
```

### 문제

- STML 생성 TSX가 `frontend/` 직하에, glue-gen 파일이 `frontend/src/`에 있어서 import 경로가 복잡해짐
- React/Vite 관례상 컴포넌트도 `src/` 아래에 있어야 함
- `src/App.tsx`에서 STML 페이지를 import할 때 `../login-page` 같은 상위 참조 발생

### 수정 제안

STML generator의 출력 경로를 `pages/` 서브디렉토리로 변경:

**방법 1: STML이 서브디렉토리 구조 지원**

```go
type GenerateOptions struct {
    SubDir string // 기본값: "" (현행), 설정 시 outDir/subDir/ 에 생성
}
```

fullend가 호출 시:
```go
stmlgenerator.Generate(pages, specsDir, outDir, stmlgenerator.GenerateOptions{
    SubDir: "src/pages",
})
```

결과:
```
frontend/
├── src/
│   ├── pages/                  # STML 생성
│   │   ├── login-page.tsx
│   │   ├── course-list-page.tsx
│   │   └── my-enrollments-page.tsx
│   ├── App.tsx                 # fullend glue-gen
│   ├── api.ts
│   └── main.tsx
├── package.json
├── vite.config.ts
└── index.html
```

**방법 2: fullend가 outDir을 `frontend/src/pages`로 지정**

STML 수정 없이 fullend만 변경:
```go
stmlOutDir := filepath.Join(artifactsDir, "frontend", "src", "pages")
stmlgenerator.Generate(pages, specsDir, stmlOutDir)
```

### 권장

**방법 2** — fullend 측에서 outDir을 조정. STML은 변경 불필요.
App.tsx의 import 경로만 fullend glue-gen에서 `./pages/login-page` 형태로 생성하면 된다.

## 5. 'use client' 디렉티브

### 현황

STML generator가 모든 TSX 파일 첫 줄에 `'use client'`를 추가한다 (`imports.go:97`).
이것은 Next.js App Router 전용 디렉티브로, Vite + React 환경에서는 불필요하다 (에러는 아니지만 불필요한 코드).

### 수정 제안

옵션으로 제어:

```go
type GenerateOptions struct {
    UseClient bool // true면 'use client' 추가 (Next.js), false면 생략 (Vite)
}
```

### 권장

**낮은 우선순위** — 있어도 Vite에서 에러가 나지 않으므로 당장은 무시 가능.

## 우선순위

| 항목 | 긴급도 | fullend 측 해결 가능 여부 |
|---|---|---|
| API 함수명 케이싱 | **높음** | O — fullend가 api 객체 export |
| API import 경로 | **높음** | △ — STML이 설정 가능하게 해주면 이상적 |
| react-query 의존성 | **높음** | △ — STML이 목록을 반환하면 이상적, 아니면 fullend 하드코딩 |
| 출력 폴더 구조 | 중간 | O — fullend가 outDir 조정 |
| 'use client' 디렉티브 | 낮음 | X — STML 수정 필요 |

## fullend 측에서 해결 가능한 항목

항목 1(케이싱), 4(폴더 구조)는 STML 수정 없이 fullend glue-gen만 수정해도 해결된다.
항목 2(import 경로), 3(의존성)은 STML이 설정 옵션/반환값을 제공해주면 가장 깔끔하지만, fullend 하드코딩으로도 우회 가능하다.

**STML 측 최소 수정 범위: 항목 2의 APIImportPath 옵션 추가만으로 충분하다.**
나머지는 fullend가 처리한다.
